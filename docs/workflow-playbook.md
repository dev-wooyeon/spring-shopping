# 과제4 학습형 워크플로우 플레이북

## 목적
기능 구현 속도보다 재현 가능한 개발 순서를 체득하는 데 집중한다.

## 기본 원칙
1. 기능 단위로만 작업한다.
2. 한 기능은 하나의 브랜치와 하나의 PR로 마무리한다.
3. 메인 브랜치 머지는 Squash Merge를 기본으로 사용한다.
4. 리뷰는 로컬 점검과 PR 리뷰를 모두 수행한다.

## 브랜치 규칙
- 패턴: `codex/t{번호}-{feature}-{action}`
- 예시: `codex/t003-product-create`

## 커밋 규칙
- 형식: `type(domain): header`
- type 예시: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`
- 본문은 2~4줄로 변경 이유와 선택 근거를 적는다.

## 기능 1개 처리 순서
1. README 작업보드에서 대상 기능을 `In Progress`로 변경한다.
2. 설계 메모를 10분 작성한다.
3. 구현을 `feat -> test -> refactor` 순서로 진행한다.
4. PR을 Draft로 올리고 Codex 리뷰를 요청한다.
5. 리뷰 포인트를 PR 템플릿의 `반`에 기록한다.
6. 최종 선택 이유를 `합`에 기록한다.
7. 테스트 통과 후 Ready로 전환한다.
8. Squash Merge 후 README 작업보드를 업데이트한다.

## 완료 기준(DoD)
1. `./gradlew test`가 통과한다.
2. 시나리오 수동 검증을 완료한다.
3. 리뷰 피드백 최소 1건 이상 반영한다.
4. 작업보드 상태/브랜치/PR 링크를 업데이트한다.

## 추상 요청 교정 프로토콜
작업 시작 전 아래 네 가지를 먼저 확인한다.
1. 목표
2. 범위
3. 완료조건
4. 제약

하나라도 비어 있으면 구현 전에 한 줄 질문으로 확정한다.
